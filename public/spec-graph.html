1 Overview

OPP Graph extends the Open Primitive Protocol with entity resolution and cross-domain traversal. The base protocol defines how a single provider exposes structured data. Graph defines how agents move between providers, following connections that exist in the real world.

A ZIP code contains water systems and hospitals. A hospital prescribes drugs. A drug has adverse events. These relationships exist whether we model them or not. OPP Graph makes them machine-readable so an agent can walk the chain without human instruction.

The extension adds three capabilities to any OPP-conformant provider:

  • Entity identifiers — a standard format for naming things across domains
  • Adjacency declarations — providers declare which entity types connect to which
  • Graph queries — agents request all connections for a given entity at a specified depth

2 Entity Identifiers

Every entity in the graph has a canonical identifier. The format is namespace:value, where the namespace identifies the authority and the value is the entity's ID within that authority.

Entity Type Format Example Authority
zip:value 5-digit ZIP code zip:10001 USPS
cms:value CMS provider ID cms:330214 CMS
fda:value Drug name (lowercase) fda:aspirin FDA
epa:value Public Water System ID epa:NY0100123 EPA
sec:value CIK number sec:0000320193 SEC
iata:value IATA airline code iata:DL IATA

Resolution rules

Identifiers are case-sensitive. fda:aspirin and fda:Aspirin are different entities. Providers must normalize to lowercase for drug names. ZIP codes retain leading zeros. CIK numbers are zero-padded to 10 digits.

New namespaces can be registered through the OPP registry. Each namespace maps to exactly one federal or international authority.

3 Adjacency Declaration

Providers declare their entity graph in the OPP manifest. The graph field lists which entity types the provider knows about and which connections it can resolve.

{
  "opp": "0.2.0",
  "name": "water-safety",
  "domain": "water",
  "graph": {
    "entities": ["zip-code", "water-system"],
    "adjacency": {
      "zip-code": ["water-system"],
      "water-system": ["zip-code"]
    }
  }
}

A hospital provider declares a broader set of connections:

{
  "opp": "0.2.0",
  "name": "hospital-quality",
  "domain": "hospitals",
  "graph": {
    "entities": ["zip-code", "hospital", "drug"],
    "adjacency": {
      "zip-code": ["hospital"],
      "hospital": ["zip-code", "drug"],
      "drug": ["hospital"]
    }
  }
}

Adjacency rules

  • Adjacency is directional. If a provider declares "zip-code": ["hospital"], it means "given a ZIP code, I can return hospitals." The reverse is a separate declaration.
  • A provider only declares adjacencies it can resolve. The water provider does not declare "zip-code": ["hospital"] even though that relationship exists — it belongs to the hospital provider.
  • Entity type names use lowercase kebab-case: zip-code, water-system, hospital, drug, demographics, weather.

4 Graph Query Endpoint

The graph query returns all connected entities for a given entity identifier at a specified traversal depth.

Request

GET /v1/graph?entity=zip:10001&depth=1
Parameter Type Required Description
entity string Yes Entity identifier in namespace:value format
depth integer No Traversal depth. Default: 1. Maximum: 3.
types string No Comma-separated entity types to filter. Returns only these connection types.

Response — depth 1

At depth 1, the response lists all directly connected entity types and their resolved IDs.

{
  "entity": "zip:10001",
  "depth": 1,
  "connections": [
    {
      "type": "water-system",
      "ids": ["epa:NY0100123"],
      "domain": "water",
      "provider": "water-safety"
    },
    {
      "type": "hospital",
      "ids": ["cms:330214", "cms:330101"],
      "domain": "hospitals",
      "provider": "hospital-quality"
    },
    {
      "type": "demographics",
      "domain": "demographics",
      "provider": "demographics"
    },
    {
      "type": "weather",
      "domain": "weather",
      "provider": "weather"
    }
  ]
}

Response — depth 2

At depth 2, the query follows each connection one step further. The hospital connection expands to include drugs prescribed at those hospitals.

{
  "entity": "zip:10001",
  "depth": 2,
  "connections": [
    {
      "type": "water-system",
      "ids": ["epa:NY0100123"],
      "domain": "water",
      "provider": "water-safety",
      "connections": []
    },
    {
      "type": "hospital",
      "ids": ["cms:330214", "cms:330101"],
      "domain": "hospitals",
      "provider": "hospital-quality",
      "connections": [
        {
          "type": "drug",
          "ids": ["fda:metformin", "fda:lisinopril", "fda:atorvastatin"],
          "domain": "drugs",
          "provider": "drug-effects"
        }
      ]
    }
  ]
}

Depth limits

Maximum depth is 3. Each depth level multiplies the number of provider calls. At depth 1, the registry resolves which providers serve the root entity type. At depth 2, the registry resolves providers for each connected entity type. Providers should set reasonable timeouts — 5 seconds per depth level.

5 Federation

Multiple OPP providers can serve data about the same entity type. Two hospital providers might both resolve zip-code to hospital. The graph query handles this through the registry.

Resolution order

  1. The agent sends GET /v1/graph?entity=zip:10001&depth=1 to the registry.
  2. The registry looks up all providers whose manifests declare adjacency from zip-code.
  3. The registry fans out requests to each provider in parallel.
  4. Responses are merged. If two providers return the same entity ID, it appears once. If they return different IDs, both appear.
  5. The merged response includes provider fields so the agent knows which provider resolved each connection.

Conflict resolution

When two providers return conflicting data for the same entity ID, the registry does not pick a winner. Both responses are included with their provider attribution. The agent decides which to trust.

Provider ownership

The registry tracks which provider is the canonical source for each entity type. The canonical provider appears first in merged results. Non-canonical providers are marked with "canonical": false.

{
  "type": "hospital",
  "ids": ["cms:330214"],
  "domain": "hospitals",
  "provider": "hospital-quality",
  "canonical": true
}

6 Use Cases

Full profile of a ZIP code

An agent receives the prompt: "Tell me everything about ZIP 10001." It sends a depth-1 graph query and gets back water systems, hospitals, demographics, and weather. Each connection includes the provider endpoint. The agent calls each provider for full data and assembles a complete profile.

GET /v1/graph?entity=zip:10001&depth=1
// Returns: water, hospitals, demographics, weather
// Agent follows each connection to its provider

Multi-hop safety query

An agent receives: "Which hospitals near 10001 prescribe drugs with more than 1000 adverse events?" This requires two hops. ZIP to hospitals, then hospitals to drugs.

GET /v1/graph?entity=zip:10001&depth=2&types=hospital,drug
// Returns: hospitals in 10001, drugs at each hospital
// Agent filters drugs by adverse event count from the drugs provider

Parallel comparison

An agent receives: "Compare the full profile of ZIP 10001 and ZIP 90210." It sends two graph queries in parallel. The results share the same structure, so the agent can diff them field by field.

// Parallel requests:
GET /v1/graph?entity=zip:10001&depth=1
GET /v1/graph?entity=zip:90210&depth=1
// Agent diffs the two response trees

Entity discovery

An agent does not know what data exists for a given entity. The graph query at depth 1 acts as a discovery mechanism. The agent learns what domains connect to that entity without prior knowledge of the OPP ecosystem.
