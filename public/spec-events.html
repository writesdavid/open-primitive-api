OPP v0.2.0 — Events Extension

Event-Driven Data

Draft — March 2026
Extension to the OPP v0.1.0 specification. This document defines how providers push data changes to subscribers in real time. Agents stay current without polling.

1 Overview

OPP v0.1.0 is request-response. A consumer asks for data. A provider returns it. That model works when the consumer knows what to ask and when to ask it. It breaks when data changes unpredictably.

Food recalls appear without warning. Water violations surface at odd intervals. Drug safety alerts publish on their own schedule. An agent that polls every hour misses the first 59 minutes. An agent that polls every minute wastes 59 out of 60 requests.

OPP Events solve this. Providers declare what events they emit. Consumers subscribe. When data changes, the provider pushes an event envelope to every subscriber. The envelope carries the same source, freshness, confidence, citations, and proof fields defined in the base spec. No new trust model. No new data format. Just a new direction: provider to consumer.

Two transports are defined: Server-Sent Events (SSE) for lightweight streaming, and webhooks for durable delivery.

2 Event Manifest

Providers declare event capabilities by adding an events object to their opp.json manifest. This tells consumers what events exist before they subscribe.

Schema

FieldTypeStatusDescription
endpoint string Required Relative path to the SSE stream endpoint (e.g., /v1/stream).
webhookEndpoint string Optional Relative path to the webhook subscription endpoint (e.g., /v1/subscribe). Omit if the provider does not support webhooks.
supported array of strings Required Event types this provider emits. MUST contain at least one of: new_record, updated_record, deleted_record, alert.
domains array of strings Required Domain IDs that emit events. Each MUST match a domains[].id in the manifest.

3 Event Envelope

Every event uses the standard OPP response envelope with additional event-specific fields. This means agents parse events and query responses with the same code. No separate model.

Event-specific fields

FieldTypeStatusDescription
event string Required Event type. One of: new_record, updated_record, deleted_record, alert.
event_id string Required Unique identifier for this event. UUIDv4 or equivalent. Used for deduplication and resumption.
event_time string Required ISO 8601 datetime of when the event was generated by the provider.
domain string Required Domain identifier. Same as the base envelope.
source string Required Upstream data source name.
source_url string Required Direct URL to the upstream data source.
freshness string Required ISO 8601 datetime of when this data was fetched from upstream.
data object Required The domain-specific payload. For deleted_record, this MUST contain at least an identifier for the deleted record.
confidence object Optional Data quality metadata. Same schema as the base envelope.
citations object Optional Source citation. Same schema as the base envelope.
proof object Optional Cryptographic proof. Same schema as the base envelope. Covers the entire event envelope excluding the proof field itself.

Event types

TypeMeaningdata payload
new_record A record appeared in the upstream source that was not there before. Full record.
updated_record An existing record changed in the upstream source. Full updated record. Providers MAY include a changes field listing which fields changed.
deleted_record A record was removed from the upstream source. Record identifier and reason (if available).
alert A high-priority condition detected by the provider. Not tied to a single record. Alert details: severity, title, description.

Example event envelope

4 SSE Transport

Server-Sent Events provide a lightweight, HTTP-native streaming connection. The consumer opens a single GET request. The provider holds it open and writes events as they occur. No WebSocket upgrade. No binary framing. Standard HTTP.

Connection

GET /v1/stream?domains=food-recalls,drug-effects HTTP/1.1
Accept: text/event-stream

The server MUST respond with:

HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
Access-Control-Allow-Origin: *

Query parameters

ParameterTypeStatusDescription
domains string Optional Comma-separated list of domain IDs to subscribe to. Omit to receive events from all domains.
events string Optional Comma-separated list of event types to receive. Omit to receive all event types.
last_event_id string Optional Resume from after this event ID. The provider SHOULD replay missed events if they are still available.

Event format

Each event follows the SSE specification (W3C). The id field maps to event_id. The event field maps to the OPP event type. The data field contains the full JSON event envelope on a single line.

id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
event: new_record
data: {"event":"new_record","event_id":"a1b2c3d4-e5f6-7890-abcd-ef1234567890","event_time":"2026-03-20T01:00:00Z","domain":"food-recalls","source":"FDA openFDA API","source_url":"https://api.fda.gov/food/enforcement.json","freshness":"2026-03-20T00:55:00Z","data":{"product":"Frozen Chicken Tenders","reason":"Undeclared allergen: milk","classification":"Class I","status":"Ongoing","date":"2026-03-19"},"confidence":{"completeness":1.0,"methodology":"Direct FDA API poll, 5-minute interval"},"citations":{"statement":"Data from the FDA openFDA enforcement database","source_name":"FDA openFDA","source_url":"https://open.fda.gov/apis/food/enforcement/","accessed":"2026-03-20T00:55:00Z","license":"US-PD"}}

Heartbeat

Providers SHOULD send a comment line every 30 seconds when no events are pending. This keeps the connection alive through proxies and load balancers.

: heartbeat 2026-03-20T01:05:30Z

Reconnection

If the connection drops, the consumer SHOULD reconnect and pass the last received event_id via the last_event_id query parameter or the standard Last-Event-ID header. The provider SHOULD buffer events for at least 5 minutes to support reconnection. Providers MAY send a retry field to suggest a reconnection interval in milliseconds.

5 Webhook Transport

Webhooks provide durable event delivery for consumers that cannot maintain a persistent connection. The consumer registers a URL. The provider POSTs event envelopes to that URL when events occur.

Subscribe

POST /v1/subscribe HTTP/1.1
Content-Type: application/json

{
  "url": "https://agent.example.com/webhook",
  "domains": ["food-recalls"],
  "events": ["new_record", "alert"],
  "secret": "whsec_a1b2c3d4e5f6..."
}

Subscribe request fields

FieldTypeStatusDescription
url string Required HTTPS URL where the provider will POST events. MUST be HTTPS.
domains array of strings Optional Domain IDs to subscribe to. Omit to receive all domains.
events array of strings Optional Event types to receive. Omit to receive all types.
secret string Optional Shared secret for HMAC-SHA256 signature verification. If provided, the provider MUST include an X-OPP-Signature header on every webhook delivery.

Subscribe response

HTTP/1.1 201 Created
Content-Type: application/json

{
  "subscription_id": "sub_9f8e7d6c5b4a3210",
  "url": "https://agent.example.com/webhook",
  "domains": ["food-recalls"],
  "events": ["new_record", "alert"],
  "created": "2026-03-20T01:00:00Z",
  "status": "active"
}

Delivery

When an event matches a subscription, the provider sends an HTTP POST to the subscriber's URL. The body is the OPP event envelope as JSON. The headers include:

HeaderValue
Content-Type application/json
X-OPP-Event The event type (e.g., new_record).
X-OPP-Event-ID The unique event ID.
X-OPP-Signature HMAC-SHA256 hex digest of the request body, keyed with the subscription secret. Present only when a secret was provided.

Delivery expectations

The consumer MUST respond with a 2xx status code within 10 seconds to acknowledge receipt. If the provider receives a non-2xx response or a timeout, it SHOULD retry with exponential backoff: 1 minute, 5 minutes, 30 minutes, 2 hours. After 5 consecutive failures, the provider MAY deactivate the subscription and SHOULD notify the consumer if a contact mechanism is available.

Unsubscribe

DELETE /v1/subscribe/sub_9f8e7d6c5b4a3210 HTTP/1.1

Returns 204 No Content on success.

6 Use Cases

Food recall monitoring

An agent subscribes to food-recalls with event type new_record. When the FDA publishes a Class I recall for contaminated peanut butter, the provider detects the new record within 5 minutes and pushes an event envelope. The agent reads the classification, checks the distribution scope, and triggers a notification to affected supply chain partners. No polling. 5-minute latency instead of 60.

Water violation alerts

A compliance agent watches water-quality for a specific ZIP code. It subscribes via webhook, filtering for alert events. When a new MCL violation appears in EPA SDWIS data for that water system, the provider fires an alert event. The agent logs the violation, cross-references the contaminant against health thresholds, and files an internal report. The human learns about it from the agent, not from checking a website.

Drug safety surveillance

A pharmacovigilance agent subscribes to drug-effects with event types new_record and alert. When FAERS data shows a spike in adverse event reports for a specific drug, the provider emits an alert. The agent compares the spike against historical baselines, assesses whether it exceeds 2 standard deviations, and escalates to a human reviewer. Continuous monitoring without continuous polling.

7 Security

SSE authentication

Providers that require authentication for SSE streams MUST accept credentials via the Authorization header on the initial GET request. Token-based authentication is preferred because SSE does not support custom headers after the initial request.

Webhook verification

Consumers SHOULD provide a shared secret when creating a webhook subscription. The provider computes HMAC-SHA256(secret, request_body) and includes the hex-encoded result in the X-OPP-Signature header. The consumer verifies the signature before processing the event. This prevents spoofed deliveries.

Webhook URL validation

Providers MUST require HTTPS for webhook URLs. Providers SHOULD verify webhook URLs by sending a challenge request before activating the subscription. The challenge is a GET request to the URL with a challenge query parameter. The consumer responds with the challenge value in the response body.

Event proof

Providers at Level 3 conformance MUST include an Ed25519 proof object on every event envelope. This allows consumers to verify events independently of the transport layer. A webhook with a valid OPP proof is trustworthy even if the consumer cannot verify the HMAC signature.

8 Conformance

The events extension adds a fourth conformance dimension. Providers claim an events conformance level independently from their base OPP level.

LevelTransportRequirements
Events L1 SSE only Valid events object in manifest. SSE endpoint returns well-formed event envelopes. Heartbeat every 30 seconds. Supports domains filter.
Events L2 SSE + Webhook All Events L1 requirements. Webhook subscribe/unsubscribe endpoints. HMAC signature support. Retry with exponential backoff on failed deliveries.
Events L3 SSE + Webhook + Proof All Events L2 requirements. Every event envelope includes an Ed25519 proof object. Supports last_event_id resumption on SSE. Buffers events for at least 5 minutes.

A provider can be Base OPP Level 2 (Cited) and Events L1 (SSE only). The levels are independent. A provider that claims Events L3 without Base Level 3 still signs event envelopes but does not sign query responses.
